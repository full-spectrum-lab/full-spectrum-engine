# v1.0 Gate Criteria

This document defines when `full-spectrum-engine` should move from the current public beta line toward a `v1.0` claim.

The intent is conservative:

> `v1.0` is not “we have many ideas”.  
> `v1.0` is “the current local-first engine contract is stable enough that external users can rely on it”.

---

## What `v1.0` should mean

If the project ships `v1.0`, it should mean all of the following are true:

- the public local-first scope is stable
- deterministic behavior is part of the contract, not a convenience
- core audit outputs are documented and consistent
- first-run experience no longer depends on private handoff knowledge
- the release boundary is explicit and enforced

`v1.0` should **not** mean:

- full four-layer recursive architecture is complete
- protocol network is live
- Cell Manifest is fully integrated
- DreamBrain is finished
- enterprise production orchestration is ready

Those are later system layers, not the minimum engine contract.

---

## Required gate areas

### 1. Runtime stability

Must be true:

- `simulate.py` works on the public sample scenarios
- seeded runs remain deterministic
- no known blocker exists in the current CLI path
- the local API starts with documented steps

### 2. Contract stability

Must be true:

- sample output structure is stable across repeated runs with the same seed
- public API request / response shapes are documented
- breaking changes are called out explicitly in release notes
- golden sample drift is intentional, reviewed, and traceable

### 3. Audit readability

Must be true:

- `RiskVector` fields are understandable without reading source code
- `Runestone` structure is documented for external reviewers
- audit list endpoints have consistent pagination and filtering behavior
- the distinction between simulation output and final business action remains explicit

### 4. Failure clarity

Must be true:

- first-run failures have a documented troubleshooting path
- common 422 / 404 / 500 API cases are documented
- storage-path failures are diagnosable without reverse-engineering the source
- delete / cleanup safety rules are documented

### 5. Public packaging coherence

Must be true:

- README, quick start, examples, troubleshooting, API reference, and release notes agree with each other
- repo cross-links point to the current `full-spectrum-lab` structure
- the release page reflects the same boundary as the repository docs

### 6. Test discipline

Must be true:

- `pytest` is green on the public branch
- deterministic tests are mandatory
- golden sample regression tests are mandatory
- public beta validation script passes on a clean local environment

---

## Suggested decision rule

Move toward `v1.0` only if all six gate areas above are green **and** there is at least one completed hardening cycle after public beta feedback.

In practical terms:

```text
v0.8 = public beta entry
v0.9 = hardening line
v1.0 = stable local-first contract
```

If v0.9 still reveals churn in the public contract, stay in `v0.x`.

---

## Recommended v1.0 scope

The recommended `v1.0` claim is narrow:

> A stable, local-first, reproducible, auditable engine for synthetic scenario inspection and internal governance validation.

That is enough.

Do not attach broader claims to `v1.0` unless the implementation actually reaches them.
