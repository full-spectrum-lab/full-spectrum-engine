## Independent black-box cases

1. Tool declaration with only ID/type succeeds.
2. Compatible declaration missing boundaries fails with `SUBJECT_REQUIRED_FIELD_MISSING`.
3. Unknown top-level field under `reject` fails; under `warn` is preserved with a warning.
4. Modified content with an old digest fails with `SUBJECT_DIGEST_MISMATCH`.
5. Same subject ID with different active declarations fails with `SUBJECT_DECLARATION_CONFLICT`.
6. `certified_reference` without enabled L4 references fails.
7. L4 references remain visible but every `subject_ref.external_effect` is `false`.
8. CLI inline JSON and file input produce the same normalized subject reference.
9. REST and CLI produce the same subject reference and structured error codes.
10. v1.0 input without a declaration remains accepted and explicitly reports subject `none`.
11. Run once with networking blocked; no external identity/certification call is attempted.
12. Confirm the Engine never emits verified/authorized/active-external facts from declaration claims.

## Required evidence from tester

- OS, Python and dependency versions;
- tested commit hash and clean-worktree proof;
- exact commands and stdout/stderr;
- per-case PASS/FAIL with fixture or request payload;
- discovered deviations with severity and reproduction;
- final recommendation: `GO`, `CONDITIONAL GO` or `NO-GO`.

The tester must not treat the developer's `158 passed` self-test as independent evidence. AI testing is advisory; project author and assigned human roles retain release authority.
