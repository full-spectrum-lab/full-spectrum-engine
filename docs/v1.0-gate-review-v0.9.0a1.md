# v1.0 Gate Review (`v0.9.0a1`)

This is the current gate review after the `v0.9.0a1` hardening pass.

It answers one question:

> Is `full-spectrum-engine` already ready to call itself `v1.0`?

Short answer:

> Not yet, but the public API/documentation surface is now close enough that `v1.0` can be judged on explicit gates instead of intuition.

## Gate summary

| Gate area | Status | Notes |
| --- | --- | --- |
| Runtime stability | ✅ Green | CLI scenarios run, deterministic seed path is established, local API startup path is documented. |
| Contract stability | ✅ Green | OpenAPI export added, request/response surface documented, golden sample drift remains reviewable. |
| Audit readability | ✅ Green | `RiskVector`, `Runestone`, API sample pack, and explainability walkthrough are now aligned. |
| Failure clarity | ✅ Green | Structured error codes are documented; troubleshooting and warning governance are explicit. |
| Public packaging coherence | ✅ Green | README, quick start, API docs, warning notes, and gate docs are now tied together. |
| Test discipline | ✅ Green | Validation script and regression tests remain mandatory. |

## Why this is still not a `v1.0` release

Even with all six gate areas green, the repository should still remain conservative.

Reasons:

1. the public contract has just been re-documented and exported
2. it still needs at least one more real-world review / feedback cycle
3. the current line is best treated as a hardened `v0.9.x` pre-v1.0 baseline

So the correct interpretation is:

```text
v0.8 = public beta entry
v0.9 = public hardening and contract stabilization
v1.0 = only after one additional review cycle confirms the contract stays stable
```

## Recommendation

Recommended next decision:

- keep the code line in `v0.9.x`
- publish the improved public contract surface
- collect one more external reviewer cycle
- then decide whether to tag `v1.0.0-rc1` or `v1.0.0`
