# v0.9 Hardening Checklist

Status: planning draft  
Applies after `v0.8.0-beta` public entry has stabilized.

## Goal

`v0.9` should harden the existing engine surface instead of expanding the conceptual scope.

The rule is simple:

> make the current engine easier to trust before making it larger.

## Priority A — contract stability

- freeze the current public CLI contract
- freeze request / response behavior for local API endpoints that are already exposed
- document stable sample inputs and expected output invariants
- keep golden samples small, readable, and versioned

## Priority B — failure clarity

- normalize error messages for malformed scenario files
- normalize exit codes for CLI validation paths
- make storage and retention failures easier to diagnose
- add troubleshooting notes for common first-run failures

## Priority C — audit usability

- improve readability of `Runestone` fields
- improve readability of `RiskVector` field explanations
- clarify when `AuditTrace` is complete vs partial
- make human-review recommendation language more consistent

## Priority D — test hardening

- keep deterministic tests mandatory
- add API contract regression tests for current public endpoints
- add storage persistence regression tests for common query paths
- add one public-beta smoke suite that mirrors first-time user behavior

## Priority E — packaging discipline

- review whether public beta files match what an external user actually needs
- keep generated artifacts out of source control unless they are intended golden samples
- verify release notes, quick start, and examples stay in sync

## Priority F — ecosystem alignment

- keep repository cross-links accurate
- keep commons visual entry aligned with engine README
- keep QPP roadmap references aligned with the current release train

## Explicit non-goals for v0.9

Do not turn `v0.9` into a theory expansion release.

Not in scope:

- full four-layer recursive architecture implementation
- full Cell Manifest integration
- protocol-network live interoperability
- DreamBrain completion
- Frequency Economy settlement
- broad enterprise product expansion

## Exit criteria

`v0.9` is complete when:

- public README, examples, tests, and release notes are coherent
- deterministic validation remains stable
- local API and storage paths are easier to reason about
- first-time external reviewers can run the engine without guessing missing steps
